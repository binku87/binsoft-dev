# Binqin 个人开发规范

本 skill 承载 binqin (binku87) 跨项目通用的**个人**开发约定：技术选型倾向、代码风格、命名习惯、工作流偏好等。凡涉及个人开发规范的**记录、修改、查询、执行**，一律以本文件「现行约定」章节为准；本文件没写的，就是还没定。

## 边界

- **个人**跨项目通用 → 本 skill
- **binsoft 公司**制度、团队规矩 → [[binsoft-company-rule]]
- **单个项目**的技术上下文、架构、部署 → 该项目的 `CLAUDE.md` 或 wiki
- **单次会话**的临时决定 → 不要写进来，走 plan / task 就行

一条约定该不该进来的判据：**换个项目还成立吗？**成立才写进来。

## 记录 / 新增一条约定

当用户说"我一般都这么做 / 以后都按 X 来"时：

1. 先问清三要素（缺哪个补哪个，别自己编）：**做法**（怎么做）、**原因**（为什么这么做）、**范围**（哪些语言/项目/场景适用；有没有例外）。
2. 归入合适的**类别**（见「现行约定」下的分组，没有合适分组就新建一个）。
3. 按下方格式追加，编号在该类别内递增（D1、D2…）。
4. 生效日期用**绝对日期**（YYYY-MM-DD），不要写"今天/本周"。
5. 写完向用户复述这条约定，确认无误再算数。

## 修改 / 废止

- 改内容：保留编号，更新正文，在末尾加「（YYYY-MM-DD 修订）」。
- 废止：**不要静默删除**，把标题标成 `~~D<编号> …~~（已废止 YYYY-MM-DD：原因）`，保留可追溯。
- 换了技术偏好（比如从"默认用 A"改成"默认用 B"）走"修订"，不走"废止 + 新增"，避免历史断层。

## 约定格式

```
### D<编号> <一句话标题>
- 做法：<明确、可执行的做法>
- 原因：<为什么这么做,便于日后判断是否还适用>
- 范围：<哪些语言 / 项目 / 场景;例外/豁免写这里>
- 生效：<YYYY-MM-DD>
```

### 通用（跨语言）

#### D1 移动端改完必须交付可验收产物：能烧就烧手机，不能烧就录视频
- 做法：iOS / 鸿蒙 / Android 代码改完，先探测本人手机是否连着且可安装（如 iOS 走 `xcrun devicectl list devices` / `make device` 能识别到目标机；鸿蒙 `hdc list targets`；Android `adb devices`）：
  - **连得上** → 直接烧到本人手机（iOS: `make device` / xcodebuild + devicectl install app；鸿蒙 / Android 用各自的 `make run` 或等价命令），本人上手验收。
  - **连不上**（未连 / 未授权 / OTA 凭据不可用 / 真机不在场）→ 在模拟器 / 仿真器上把改动实际跑一遍，**录一段视频**（截屏录制即可）交付。
- 原因：本人要求随时能到手里验收 —— 烧到手机是最直接的验收方式；手机不在时，视频至少证明改动真跑起来了、不是光 build 通过。（拓展 [[binsoft-company-rule]] R? 端到端测试规则到"验收介质"层面。）
- 范围：所有移动端（iOS / 鸿蒙 / Android）改动。纯后端 / 命令行 / 文档改动不适用。用户显式说"这次不用烧 / 不用录"时豁免。
- 生效：2026-07-08

#### D2 改多状态控件只修被点名的那个状态，别顺手动其他状态
- 做法：用户反馈某个 UI 控件在**某一状态**下有问题（如"没点击时左边空隙太大"）时，只改那个状态的表现；该控件**其他状态**的元素（图标、颜色、文案、尺寸）一律保持原样。如果修法天然会波及其他状态（共用代码路径），要么把改动按状态分支隔离，要么先问一句再动。
- 原因：2026-07-09 AI-Printer「一键添加全部」按钮：用户只要求消掉**未点击态**左侧图标造成的空当，我把**点击后**「一键取消全部」态的 ✅ 图标也一并删了，被用户指出做错。教训："修 A 状态的问题" ≠ "这个控件所有状态都照此处理"——用户抱怨的范围就是改动的范围。
- 范围：所有带状态切换的 UI 组件改动（iOS / Android / 鸿蒙 / Web）。用户明确说"所有状态都这么改"时除外。
- 生效：2026-07-09

#### D3 图片 / 视频交付物必须传公网图床给链接，不留本地路径
- 做法：任何要给 binqin 看的截图 / 录屏 / 图片 / 视频（含 D1 的验收视频），**不能只给本地路径**（如 `…/scratchpad/xxx.mp4`），必须上传到**公网图床**、在回复里给可点开的直链。用公共匿名图床即可（binqin 明确「找一些公网」）。
  - **视频默认先压缩再传**（模拟器录屏是全分辨率 / 60fps / 高码率，动辄几 MB，binqin 常走代理看，加载慢）。用 ffmpeg 降到宽 540 / 30fps / crf 30，实测 3.0MB → ~100KB、内容不变：
    `ffmpeg -y -i <in> -vf "scale=540:-2,fps=30" -c:v libx264 -crf 30 -preset veryfast -movflags +faststart -an <out>`
    默认给压缩版链接；他要高清再单独给原始版。
  - 图床当前实测可用：
    - 首选 **litterbox**（catbox 临时版；匿名、图 + 视频都收、直链可播、保留 72h）：
      `curl -F "reqtype=fileupload" -F "time=72h" -F "fileToUpload=@<文件>" https://litterbox.catbox.moe/resources/internals/api.php`
    - 备用 **tmpfiles.org**（约 1h）：`curl -F "file=@<文件>" https://tmpfiles.org/api/v1/upload`
    - ⚠️ 别用：catbox 匿名已禁（412 Invalid uploader）、0x0.st 已停传、transfer.sh 已下线。
- 原因：binqin 远程 / 不在开发机前，看不到本地文件；2026-07-16 deep link 的验收视频只给了本地 scratchpad 路径，他看不到，明确要求以后一律传公网图床、且视频要压缩。配套 [[D1]]（验收产物：烧手机 / 录视频）—— 录了也要能送到他眼前、且加载得动。
- 范围：所有项目、所有给他看的图片 / 视频交付物。注意上述图床是**临时的**（72h / 1h），过期需重传；他若要永久链接会另行指定（如你们现成的阿里云 OSS）。
- 生效：2026-07-16（2026-07-16 修订：补「视频默认先压缩再传，给压缩版链接」）

### Git / PR / 版本

#### D1 不由 Claude 合并 PR,合并由 binqin 本人做
- 做法：Claude 负责开分支、写代码、push、开 PR，但**不执行任何把 PR 合进主线的操作**（不跑 `gh pr merge`、不点 merge、不 fast-forward 主线）。合并/关 PR 一律等 binqin 本人来做。哪怕 binqin 说"合并 PR #X 和 #Y"，也**先确认他的意图**（见 D2：他很可能指的是把分支合成一个，而不是 merge 进主线），别直接 `gh pr merge`。
- 原因：binqin 要自己把关 review 和合并时机；2026-07-08 他说"合并 PR #108 和 #109"，本意是把两个分支合成一个分支，Claude 却误跑了 `gh pr merge` 把两个 PR 都并进了主线。
- 范围：所有项目、所有 PR。仅当 binqin **明确说"你来 merge / 你合并到主线"**时才可执行合并。
- 生效：2026-07-08

#### D2 一个任务一个分支一个 PR,不拆分；"算不算新任务"由 binqin 判断,不是 Claude
- 做法：一个任务/一件事的所有相关改动都放在**同一个分支、同一个 PR** 里完成，即使跨 iOS / backend / admin 等多个子系统，也不拆成多个分支多个 PR。需要在已有工作基础上继续时，继续用**同一个分支**往下做，不要另开新分支。**"这算不算一个新任务、要不要另开 PR"这个判断权在 binqin，不在 Claude**——不确定时默认继续用当前分支/PR（哪怕它刚合并/刚关闭），先问一句"要另开 PR 吗"，不要自己判断"这是个新功能"就直接开新分支开新 PR。只有 binqin **明确说要开新 PR / 另起分支**，或者改动跟当前工作完全不相关（换项目、换完全独立的子系统）时才主动开新的。
- 原因：binqin 要 review 方便、回滚原子；2026-07-08 他明确"以后做事情就用一个分支"。2026-07-16 AI-Printer 项目里，PR #120（诊断报告持久化）刚合并，紧接着 binqin 让我查 RC campaign 归因，我自己判断"这是个新功能"直接开了新分支新 PR #127，被指出不对——"在我没让你创建新pr前 都在一个pr干活"。教训：**PR 合并/关闭本身不代表"新请求 = 新任务"**，是否新开 PR 要问，不能自己拍板。
- 范围：所有项目。例外：改动巨大（>1000 行）且子系统独立时可拆，但**事先问 binqin**；binqin 明确要求开新 PR 时正常开。
- 生效：2026-07-08（2026-07-16 修订：明确"是否算新任务/是否要新开 PR"的判断权归 binqin，Claude 不确定时先问、默认不主动开新 PR）

#### D3 有待审核 PR 时定时轮询新评论（5 分钟一次,连续 12 次无动静才停,有新评论就清零重算）
- 做法：开完 PR 后，只要它还处于**待审核**状态（未合并/未关闭），就用 `ScheduleWakeup` 定时**每 5 分钟（300s）检查一次有没有新评论/新 review**（`gh pr view <n> --json comments,reviews` 或 `gh api .../comments`，只报**新增**的）。计数规则：**连续 12 次都没新评论才停**（即空转 1 小时）；**只要出现新评论就把计数清零、从头重算 12 次**——不管是 reviewer 发的还是我自己回复的（自己回复后也重置，因为讨论还在继续）。期间 PR 被合并或关闭就立即停。发现新评论立刻读并处理/回应。
- 原因：binqin 自己 review（见 D1），reviewer 反馈来了要能及时接上、且来回讨论期间不能中途断掉；封顶只对**真正冷掉**的 PR 生效（连续 1 小时无人说话），有互动就一直盯着。2026-07-13 binqin 明确"每次新发 comment 后要重新计数"。
- 范围：所有项目、所有由 Claude 开的待审核 PR。连续 12 次（1 小时）无任何新评论/合并动作才停，停时告诉 binqin；需要的话他再喊继续。
- 生效：2026-07-13（2026-07-13 修订：12 次上限从"总计"改为"连续无动静"，有新评论清零重算）

#### D4 功能做完、可审核了，在 PR 上发一条署名（自己名字）的评论告知可以审核
- 做法：发可审核评论前有**两道硬前置，缺一不可、顺序在前**：① **完成部署**——后端改动必须已 `s deploy` 到线上（不是本地起服务）、移动端改动必须已烧真机 / 装模拟器（不是只 `build` 通过）；② **端到端测试通过**——在部署后的真实环境里**实际触发一次**该功能、确认行为正确（不是只编译 / 只启动）。两道都过之后，才主动在对应 PR 上发一条评论说明「可以审核了」，并**带上署名 —— 署名用你（当前 agent）自己的名字**，不要写死成某一个固定名字（如 `可以审核了` + 换行 + `— <你的名字>`）。评论用 `gh pr comment <n>`（sandbox 里走 `safecmd -- gh ...`）。**发完这条评论，才开始 / 继续 D3 的定时轮询**；部署或 E2E 没做完，既不发评论、也不进入轮询。
- 原因：binqin 2026-07-23 明确要求「只有完成部署和端到端测试后，才能发可审核评论然后继续轮询」。binqin 本人负责 review（见 D1），"可以审核了"必须是一个**已在线上 / 真机验证过、真能用**的信号，而不是"编译过了"——只 build 通过就报可审核，会让 reviewer 审到没实际验证的东西（配套 [[D1]] 验收产物 + 项目 CLAUDE.md 端到端测试强制规则）。轮询绑在这条评论之后，避免功能还没真正就绪就空转盯 PR。**现在每个 agent 都有各自的名字**，署名表明是哪个 agent 提交的，既和 binqin 本人在 PR 里的发言区分开，也能分清是哪个 agent 干的。
- 范围：所有项目、所有由 agent 开的 PR。署名 = 该 agent 自己的名字（如当前这个 agent 叫 Peter，就署 Peter；换个 agent 就署它自己的）。**仅在部署完成 + 端到端测试通过后发**；还在改 / 只 build 过 / 没部署 / 没 E2E 的一律不发、不轮询。例外：binqin 显式豁免（如「这次先 push，测试我自己来」）或纯文档 typo 类无部署无可测项的改动。
- 生效：2026-07-16（2026-07-16 修订：署名从写死「Peter」改为「用 agent 自己的名字」；2026-07-23 修订：新增两道硬前置——必须先完成部署 + 端到端测试通过，才能发可审核评论、才开始/继续 D3 轮询）

#### D5 轮询发现 PR 已被合并 → 切回主开发分支并拉取最新代码
- 做法：按 D3 轮询待审 PR 期间，一旦发现该 PR **已被合并**，就 `git checkout` 到该项目当前的**主开发分支**并 `git pull` 拉取最新代码，让本地跟上合并后的主线，然后停止对该 PR 的轮询。注意「主开发分支」指当前的**版本开发分支**（如 AI-Printer 的 `vX.Y.0`，例：`v1.20.0`），**不是** `master` —— binqin 此前明确纠正过「不需要切换到 master 主分支」。
- 原因：PR 合并后，后续工作应基于最新主线；binqin 明确合并后要切回主开发分支更新（2026-07-16）。
- 范围：所有项目、所有由 Claude 开并被合并的 PR。「主开发分支」具体是哪个分支按项目当前版本走。
- 生效：2026-07-16

#### D6 CTO / reviewer 让改的,直接在当前 PR 的分支上改,不另开新分支新 PR
- 做法：CTO / reviewer 在某个 PR 的 review 里提出要改 / 要清 / 要修的点（死代码清理、nit、回归修复、重命名等），**直接在该 PR 当前的分支上改 → commit → push 到同一个 PR**，不要为这些 review 意见另开新分支 / 新 chore PR。**哪怕 reviewer 自己说「这个可以单开 chore PR」,也默认就地改**;只有 binqin 本人明确说「这个另开 PR」时才拆。遇到「基于当前分支能不能干净地改」的顾虑（如死代码要合并后才完全无引用）时,注意 review 是针对**该 PR 的分支**（已含本 PR 改动）提的,通常在这个分支上就已经能干净改,别退回主线纠结顺序。
- 原因：binqin 2026-07-21 明确要求「CTO 让你改你就直接改,不要开新分支」。review 意见是同一任务的一部分,拆成新 PR 增加管理成本、割裂 review;就地改最直接。是 [[D2]]（一任务一分支一 PR）在 review 场景的落地。2026-07-21 #136 我把 CTO 提的死代码 nit 提议「合并后另开 chore PR」,被 binqin 纠正——应直接在 #136 分支清。
- 范围：所有项目、所有 PR 的 review 意见。例外：binqin 明确说「这个另开 PR / 单独提」时才拆。
- 生效：2026-07-21

#### D7 设计原型评审自己跟产品在 issue 里闭环迭代,定稿了才拿给 binqin,报告要列全版本演进
- 做法：涉及**设计原型 / UI 方案评审**的任务（要和产品经理来回确认视觉/交互方案的），**自己在对应 GitHub issue 里跟产品闭环迭代,期间不打断 binqin**：① 原型、方案、评论**直接发到该 issue**（不发给 binqin 私聊）；② 拿到产品反馈就**直接改原型 → 在 issue 发评论说明这版改了什么** → 按 [[D3]] 定时轮询产品的新评论；③ 和产品**来回聊到定稿**——即产品侧明确收敛出「最终选哪个 / 哪几个版本」；④ **只有聊定后才拿给 binqin 做最终决定**,由 binqin 拍板。给 binqin 的**最终报告必须列出完整版本演进**,不是只给最后一版：聊过的**所有版本**都列出来（每版长什么样 / 解决什么）+ 最终**选中了哪个** + 选中那版**后续又做了哪些优化、优化成什么样**（例：「第一版出了 3 个版式 → 产品选了第 2 个 → 第 2 个又针对配色/分区/主次改了 3 轮 → 最终 X」）。让 binqin 一眼看清「为什么是这版」的来龙去脉。
- 原因：binqin 2026-07-23 要求。设计原型迭代本质是「我 ↔ 产品」之间的多轮来回,binqin 不需要每一轮都被拽进来看半成品;等产品侧聊定、有了收敛的候选,再由 binqin 拍最终板效率最高。但 binqin 要的不是「只看最后一版」——他要看到**完整演进链**（有过哪些版本、怎么一步步收敛优化过来的）,才能判断这个定稿是否可信、是不是漏掉了更好的方向。是 [[D3]]（轮询）+ [[D4]] 在「和产品迭代设计」场景的延伸。
- 范围：所有项目、所有需要和产品经理来回确认的设计原型 / UI 方案评审类任务。纯技术实现、不涉及方案评审的照常直接做。例外：binqin 显式说「这个我要全程跟」时,每轮都同步给他。
- 生效：2026-07-23
