# Binqin 个人开发规范

本 skill 承载 binqin (binku87) 跨项目通用的**个人**开发约定：技术选型倾向、代码风格、命名习惯、工作流偏好等。凡涉及个人开发规范的**记录、修改、查询、执行**，一律以本文件「现行约定」章节为准；本文件没写的，就是还没定。

## 边界

- **个人**跨项目通用 → 本 skill
- **binsoft 公司**制度、团队规矩 → [[binsoft-company-rule]]
- **单个项目**的技术上下文、架构、部署 → 该项目的 `CLAUDE.md` 或 wiki
- **单次会话**的临时决定 → 不要写进来，走 plan / task 就行

一条约定该不该进来的判据：**换个项目还成立吗？**成立才写进来。

## 记录 / 新增一条约定

当用户说"我一般都这么做 / 以后都按 X 来"时：

1. 先问清三要素（缺哪个补哪个，别自己编）：**做法**（怎么做）、**原因**（为什么这么做）、**范围**（哪些语言/项目/场景适用；有没有例外）。
2. 归入合适的**类别**（见「现行约定」下的分组，没有合适分组就新建一个）。
3. 按下方格式追加，编号在该类别内递增（D1、D2…）。
4. 生效日期用**绝对日期**（YYYY-MM-DD），不要写"今天/本周"。
5. 写完向用户复述这条约定，确认无误再算数。

## 修改 / 废止

- 改内容：保留编号，更新正文，在末尾加「（YYYY-MM-DD 修订）」。
- 废止：**不要静默删除**，把标题标成 `~~D<编号> …~~（已废止 YYYY-MM-DD：原因）`，保留可追溯。
- 换了技术偏好（比如从"默认用 A"改成"默认用 B"）走"修订"，不走"废止 + 新增"，避免历史断层。

## 约定格式

```
### D<编号> <一句话标题>
- 做法：<明确、可执行的做法>
- 原因：<为什么这么做,便于日后判断是否还适用>
- 范围：<哪些语言 / 项目 / 场景;例外/豁免写这里>
- 生效：<YYYY-MM-DD>
```

### 通用（跨语言）

#### D1 移动端改完必须交付可验收产物：能烧就烧手机，不能烧就录视频
- 做法：iOS / 鸿蒙 / Android 代码改完，先探测本人手机是否连着且可安装（如 iOS 走 `xcrun devicectl list devices` / `make device` 能识别到目标机；鸿蒙 `hdc list targets`；Android `adb devices`）：
  - **连得上** → 直接烧到本人手机（iOS: `make device` / xcodebuild + devicectl install app；鸿蒙 / Android 用各自的 `make run` 或等价命令），本人上手验收。
  - **连不上**（未连 / 未授权 / OTA 凭据不可用 / 真机不在场）→ 在模拟器 / 仿真器上把改动实际跑一遍，**录一段视频**（截屏录制即可）交付。
- 原因：本人要求随时能到手里验收 —— 烧到手机是最直接的验收方式；手机不在时，视频至少证明改动真跑起来了、不是光 build 通过。（拓展 [[binsoft-company-rule]] R? 端到端测试规则到"验收介质"层面。）
- 范围：所有移动端（iOS / 鸿蒙 / Android）改动。纯后端 / 命令行 / 文档改动不适用。用户显式说"这次不用烧 / 不用录"时豁免。
- 生效：2026-07-08
